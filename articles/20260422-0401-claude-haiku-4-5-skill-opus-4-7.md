もう Opus を常用するのはやめた。

正確に言う。やめられた、が正しい。
skill を書いてから、Haiku 4.5 で足りるタスクが一気に増えた。

きっかけは Reddit の r/ClaudeAI に上がっていた検証スレだ。9モデル × skill 有無のマトリクスで回した結果、Haiku 4.5 + skill が baseline の Opus 4.7（素の状態）を上回った ― というやつ。最初に見たときは正直「またベンチ職人の都合いい数字か」と思った。思ったが、自分の Claude Code 環境で追試したら、傾向が出た。出てしまった。

この記事は、その数字の意味を Claude Code で skill/hook/MCP を日常運用している立場から分解する話だ。

## このノートで分かること

- Reddit 検証スレの 9モデル × skill マトリクスが示した本当の意味
- skill を挟むと小型モデルが伸びる「3つのメカニズム」
- 僕の Claude Code 環境での再現検証結果（skill あり/なしで比較）
- Haiku / Sonnet / Opus をどのタスク層に割り当てるかの実運用ルーティング
- 脳過敏症エンジニアが Opus 常用から降りて得た「思考コスト」の実数

## 元スレの数字を冷静に要約する

元スレ（[Reddit: tested 9 models with and without agent skills](https://reddit.com/r/ClaudeAI/comments/1srpv7c/tested_9_models_with_and_without_agent_skills/)）の主張はシンプルだ。

9モデルを、同一タスクで skill 有り / skill 無しの2条件で回した。結果、**Haiku 4.5 + skill が、素の Opus 4.7 より高スコア**を出した条件がある。

ここで大事なのは「全タスクで超えた」ではない点だ。投稿者自身も「条件依存」と注釈している。ただし、手順が固定化できるタイプのタスク（リファクタ、テスト生成、CRUD 実装、整形系）では、Haiku + skill が安定して Opus を食っていた。

コスト差は Haiku 4.5 と Opus 4.7 で概ね 1/15 前後。レイテンシ差は 1/3 前後。（※モデル単価とトークン量の公開情報からの概算。タスク毎にブレる）

この時点で僕の頭の中では計算が終わっていた。
**skill を書く時間 < 節約できるAPIコスト × 発火回数**、なら書かない理由がない。

## なぜ skill で小型モデルが伸びるのか ― 3つのメカニズム

ブラックボックスで「skill すごい」と言うのは気持ち悪いので、分解する。

**①プロンプト圧縮**
skill は手順をコードに落としたものだ。毎回 Opus に「こういう流れで、こういう観点で、こう出力して」と自然言語で渡すのと、skill として固定された手順を呼ぶのでは、モデルが使う「推論の持ち札」の枚数が違う。Haiku は推論力で劣るが、手札を事前に配ってあれば負けない。

**②手順固定**
Opus の強みは「曖昧な指示から最適手順を自分で組み立てる」ところにある。逆に言えば、手順が決まっているタスクではそのパワーは遊ぶ。skill は手順を固定するので、Opus の優位が消える。

**③評価基準の明示**
skill には「何を満たしたら完了か」が書ける。Haiku は自分で評価軸を組み立てるのは苦手だが、渡された軸で採点するのは得意だ。ここで差が埋まる。

簡単に言うと、skill は**モデルの推論力を補助輪で代替する装置**だ。補助輪があれば、ママチャリ（Haiku）でもロードバイク（Opus）と同じ舗装路を走れる。

ただし ― 補助輪は悪路では外れる。未知ドメインの設計判断、トレードオフの多い意思決定、長い文脈をまたぐ一貫性維持。ここは今でも Opus が強い。

## 再現検証: 自分の brain 環境で回してみた

僕は Claude Code で brain という自律AIシステムを自作して、夜間自走させている。skill は今 40個くらい、hook は 12個、MCP は 5個。日常的にモデルを差し替えて回している環境だ。

今回、手元で以下3タスクを skill 有り / 無し × Haiku 4.5 / Opus 4.7 の4条件で回した。

- タスクA: 既存TSファイルへのテスト生成（Vitest）
- タスクB: note 記事の構成案生成（自作 note-pipeline skill）
- タスクC: autoresearch の収集データから記事候補抽出

結果は ― 正直に言うと、元スレほどキレイな逆転は出なかった。

タスクA（テスト生成）は Haiku + skill ≒ Opus（素）。体感で並ぶ。
タスクB（構成案）は Haiku + skill > Opus（素）。skill に僕の過去266記事の文体ルールが入っているので、Opus の一般論より噛み合う。
タスクC（候補抽出）は Opus（素）> Haiku + skill。ここは文脈判断が多く、Haiku は取りこぼす。

10回ずつ回した雑な試行なので、統計的に有意とまでは言わない（未検証）。
ただ**傾向は明確に出た**。手順固定タスクは Haiku で足りる。判断タスクは Opus が要る。

## 実運用モデルルーティングの設計図

じゃあどう割り振るか。僕の brain の model-route skill では、ざっくり以下の層分けにしている。

**Haiku 4.5 層（skill 必須）**
- 定型の実装（CRUD、ボイラープレート、テスト生成）
- 文体固定の文章生成（note-pipeline 経由の記事下書き）
- 構造化データの抽出・整形
- hook からの短命タスク

**Sonnet 4.6 層（skill 推奨）**
- 中程度の設計判断を含む実装
- コードレビュー
- 複数ファイルをまたぐリファクタ

**Opus 4.7 層（skill 任意）**
- アーキテクチャ設計
- 新規ドメインの調査・要件整理
- 長い文脈の一貫性が要るタスク

常識を壊しておく。
「最上位モデルを常用すれば安全」― 嘘。
「Haiku は品質が落ちる」― 条件付きで嘘。skill があれば埋まる。
「モデル選定が先」― 嘘。**skill 設計が先**。

ただし、これは僕の brain という「手順が固まってきた運用」での話だ。立ち上げ期はむしろ Opus で手順を発見し、それを skill に落とす流れの方が早い。スタートアップと運用フェーズで最適解が違う。

## 脳過敏症エンジニアの「疲労コスト」

ここだけ技術から離れる。

僕は脳過敏症を持っていて、長時間の集中が効かない。Opus を常用していた頃、コード生成は早かったが、**出力を読み切る方が疲れた**。Opus は気を利かせすぎる。頼んでない提案を混ぜる。文脈を汲んで別案を並べる。ありがたい、が、読むコストが重い。

Haiku + skill に降ろしたら、出力が短くなった。skill で「出力フォーマットはこれ」と縛っているからだ。

読む。判断する。次へ行く。

この3ステップが一瞬で終わるようになって、1日に回せる実験回数が増えた。APIコストが 1/15 になったのは副次効果で、本当のリターンは**自分の脳の消費量が減った**ことだった。

正直に言う。これはベンチには出ない。出ないが、僕にとっては一番効いた変数だ。

## まとめ: skill はモデル選定より一階層上の設計

整理する。

- Haiku 4.5 + skill が素の Opus 4.7 を超えるのは、手順固定タスクに限る
- メカニズムは「プロンプト圧縮 / 手順固定 / 評価基準明示」の3点
- 実運用ではタスク層で割り振る、skill がない領域では Opus が今も強い
- モデル選定より skill 設計を先にやる方が効く

「どのモデル使えばいいですか」と聞かれたら、最近の僕は「先に skill 書いてください」と返している。モデルは skill が決まった後にハマる場所が見える。逆はない。

ただし ― skill を書くにはまず Opus で手順を発見するフェーズが要る。鶏と卵だが、発見は Opus、運用は Haiku、という二段構えで考えると迷わない。
