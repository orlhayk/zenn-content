6000行のPRを Claude Code に書かせた。レビュー観点42項目のうち、「認可」に関する指摘はゼロ件だった。生成側のClaudeも、監査側のAIレビュアーも、誰一人として「この関数、誰が呼べるの？」を問わなかった。

正直に言う。これは僕の設計ミスだ。AIのせいじゃない。

元記事は Qiita の [@nrEngineer さんのポスト](https://qiita.com/nrEngineer/items/1ce21b74d887e8de68cc)。like=0 / stock=1 の地味エントリだが、読んで震えた。僕が brain (自律AI) に夜間コードを書かせてる立場で、まさに踏みかけてた地雷だったからだ。

今日はこの事件を、Claude Code の skill / hook / MCP を触ってる側の視点で再解釈する。

## このノートで分かること

- AIが認可を書き飛ばす10個の地雷を「生成由来／レビュー由来／仕様由来」の3分類で整理
- なぜAIレビュアー（PR-Agent, CodeRabbit, Claude Code Review）まで揃えても認可を見逃すのか
- 認可を「プロンプトでお願いする」から「hook で強制する」へ構造を組み替える方法
- 監査用サブエージェント `authz-reviewer` の最小設計
- 明日から使える 7項目チェックリスト

結論を先に書く。**認可はプロンプトに書いてはいけない。** 書いた瞬間、それは祈りになる。

## 10個の地雷を3分類で整理する

元記事の列挙を、僕の運用視点で分類し直した。

**生成由来（AIが書き落とす）4件**
- エンドポイントに認可デコレータを付け忘れる
- ORM の findById を authUser スコープなしで叩く
- 管理画面の画面出し分けを `if role === 'admin'` だけで済ませAPIを素通り
- バッチ処理・内部API・webhook ハンドラに認可レイヤを入れない

**レビュー由来（AIが指摘し損ねる）4件**
- 差分内で整合してるとOK判定してしまう（横断検査をしない）
- 「既存と同じパターンです」で既存のバグごと承認
- テストが通ってるから認可も通ってる、と誤読
- 認可の有無をコードから推論するだけで、ロール表と突き合わせない

**仕様由来（そもそも書かれてない）2件**
- ロール定義が PRD になく、実装者が「たぶんこう」で補間
- 「誰が何をできるか」の表が1枚もないまま実装開始

分けてみて分かる。**生成AIが悪いんじゃない。仕様を渡してないから補間されてるだけだ。**

## なぜAIレビュアーも見逃すのか

僕は Claude Code Review と自作の `code-reviewer` サブエージェントを二段で噛ませてる。それでも認可は抜ける。理由は1つ。

**AIレビュアーは差分しか見ない。**

PRの diff には「認可が必要だったはずの関数」が映らない。付いてない行は diff に出ない。出ないものは指摘できない。当たり前の話だが、これがAIレビューの構造的欠陥だ。

さらに悪いことに、生成Claudeと監査Claudeは同じ学習分布から答えを引く。生成側が「これくらい書けば十分」と判断した箇所は、監査側も「十分に見える」と判断する確率が高い。**同じ盲点を持つ二人に同じ問題を解かせても、盲点は埋まらない。**

僕はこれを痛感してから方針を変えた。

## Claude Code hook で認可を「構造」として強制する

プロンプトに「認可を忘れずに」と書いても意味がない。断言する。

祈りだ。設計じゃない。

僕が brain に組み込んだのはこれだ。

```
PostToolUse hook:
  trigger: Edit/Write が *.ts, *.py に対して発火
  action:  変更ファイルに対し authz-reviewer サブエージェントを起動
  block:   認可マトリクス未参照の関数が1つでもあれば PR 作成をブロック
```

ポイントは「PR作成を物理的に止める」こと。AIが「忘れた」ときに気づくのではなく、**忘れたままマージ動線に入れない**ようにする。

認可マトリクスは YAML で1枚持っておく。

```yaml
# authz-matrix.yaml
resources:
  admin_dashboard:
    read: [admin, owner]
    write: [owner]
  user_profile:
    read: [self, admin]
    write: [self]
```

hook が起動する `authz-reviewer` は、このYAMLと変更コードを MCP 経由で突き合わせる。diff だけを見るのではなく、**「変更関数が認可マトリクスのどの行に対応するか」を必ず1対1で紐付けさせる**。紐付かない関数があれば赤信号。

## 監査用サブエージェント `authz-reviewer` の最小設計

僕が実装した最小版はこう。

- **入力**: PR diff + authz-matrix.yaml + 既存エンドポイント一覧
- **処理1**: 新規/変更関数を列挙
- **処理2**: 各関数を authz-matrix の resource × action にマッピング
- **処理3**: マッピング不能な関数をリスト化
- **出力**: 未マッピング関数ゼロ件でのみ GREEN、1件でもあれば RED

モデルは Haiku 4.5 で十分だった。認可チェックは「推論」より「突き合わせ」の仕事だからだ。Opus を回すと逆に過剰解釈して誤検知が増えた。未検証だが、この傾向は多くのチェック系タスクで共通しそうだ。ただし機密リソース定義が複雑なドメインでは Sonnet 以上が無難だと思う。

## 明日から使えるチェックリスト（7項目）

俺が 6000行PR 事件の反省で作った実運用版だ。

1. 認可マトリクス（YAML 1枚）を PRD 段階で確定させる
2. 新規エンドポイントは「誰が呼べるか」をコメントで1行明記
3. PostToolUse hook で authz-reviewer を強制発火
4. 生成Claudeと監査Claudeは別プロンプト・別モデルで分離
5. diff外ファイルの横断検査を MCP で担保
6. バッチ・webhook・内部APIも例外なくレイヤを通す
7. ロール表と実装の対応を週次で棚卸し

やってみて分かったが、**認可は「書く技術」より「書き忘れない構造」の問題**だ。AIが書こうが人間が書こうが、構造で縛らないと抜ける。

## 終わりに

AIに大きなPRを任せ始めた人ほど、認可だけは「AIにお願いする」のをやめた方がいい。プロンプトは祈り、hook は契約だ。僕は brain を夜間自走させてるが、認可チェックだけは hook で物理的に止めるようにしてから事故率が一気に落ちた。

同じことやってる人いたら、authz-matrix の YAML スキーマをどう切ってるか教えてほしい。特に SaaS のマルチテナントでの resource 粒度は、まだ僕の中でも正解が見えてない。

試す入り口として、Claude Code の hook 公式ドキュメント（[docs.anthropic.com/claude-code](https://docs.claude.com/)）と、元記事の Qiita ポストを並べて読むのがおすすめだ。地雷の具体例と、それを踏まない構造の両方が手に入る。
