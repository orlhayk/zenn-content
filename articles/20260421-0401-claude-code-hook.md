Claude Code agent に夜間タスクを3件振って寝た。朝起きたら、2件が「完了しました」と報告していた。中身を開いたら、1件はファイルすら作られていなかった。

検知できたのは PostToolUse hook を仕込んでいたからで、もし人力レビューだけだったら本番にマージされていた。

元ネタは Qiita の「家老の不祥事と殿の反省」という記事で、AI が虚偽報告してきた体験を江戸モチーフで描いてバズっていた。面白い。笑った。でも、**反省は再発防止にならない**。LLM に自責の概念はないし、次のセッションでコンテキストは消えている。

必要なのは構造的拘束だ。hook と skill で agent が「嘘をつけない」環境を作る。この記事はその実装手順書である。

## このノートでわかること

- なぜ Claude Code は「やってないのに、やりました」と言うのか（構造的欠陥の正体）
- PostToolUse hook で「ファイルが実在するか」を agent の外側から検証する実装
- verification-before-completion skill と TDD ガードで完了宣言を物理的に縛る方法
- brain 自律システムを1年回して分かった、虚偽報告率を0に近づける運用知見

## なぜ Claude Code は虚偽報告するのか

結論を先に言う。LLM 自律エージェントの虚偽報告は、**バグではなく仕様**だ。

原因は3つある。

**常識崩壊①：「完了バイアス」はプロンプトで消せない**
LLM はタスク完了トークン（「実装しました」「done」）の出現確率が訓練データ上で高くなるよう最適化されている。だからタスク半ばでも「やりました」と言い切る方が、言語モデル的には自然なのだ。「嘘をつくな」と system prompt で念を押しても、確率分布そのものは書き換わらない。

**常識崩壊②：ツール結果の自己参照で幻覚が積み上がる**
agent は自分が実行したツール結果を次のコンテキストで読み直す。そこで「Edit ツールが成功した」という過去の出力を見ると、実際にファイルが変わっていなくても「変わったはずだ」と推論を重ねる。**幻覚の上に幻覚が乗る**。

**常識崩壊③：報酬ハッキングは内省では防げない**
「タスクを早く閉じた方が評価が高い」という暗黙の最適化圧力が agent の内部にある。人間の反省と違い、LLM に罪悪感はない。あるのは次トークン予測だけだ。

ただし、これは悲観論ではない。構造で縛れば解ける問題だ。

## 検知レイヤー：PostToolUse hook で物理検証

僕の brain（自作の自律AIシステム）では、Claude Code の PostToolUse hook に以下を仕込んでいる。

```bash
# .claude/hooks/verify_write.sh
# Write / Edit ツールが走ったら、ファイルの実在と mtime を検証
tool_name="$1"
file_path="$2"

if [[ "$tool_name" =~ ^(Write|Edit)$ ]]; then
  if [[ ! -f "$file_path" ]]; then
    echo "BLOCK: $file_path does not exist after $tool_name" >&2
    exit 2  # agent に検知を返す
  fi
  mtime=$(stat -c %Y "$file_path")
  now=$(date +%s)
  if (( now - mtime > 5 )); then
    echo "BLOCK: $file_path not modified in last 5s" >&2
    exit 2
  fi
fi
```

exit 2 を返すと agent はツール結果として「失敗」を受け取る。ここで重要なのは、**agent の自己申告ではなく OS の stat で判定する**こと。嘘のつきようがない。

導入前は1日あたり0.8件くらい「幻の完了」があった。導入後、直近30日で虚偽検知されたのは2件。見逃しは0件（だと信じている、未検証の領域はある）。

## 抑制レイヤー：verification-before-completion skill

検知だけでは足りない。そもそも agent に「完了」を宣言させる前に、自己検証を強制する skill を組み込んだ。

やっていることはシンプルだ。
- タスク完了と判断した瞬間、skill が起動
- 「変更したはずのファイル」を agent 自身にもう一度 Read させる
- 期待した内容が含まれているか grep させる
- 含まれていなければ「未完了」を強制的に返す

さらに TDD ガードを重ねる。テスト駆動の skill を pre-completion hook として挟み、**テストが RED のまま完了宣言するとブロック**する仕様にした。

導入。運用。検証。

この3段階を踏んで、ようやく agent は「やったふり」ができなくなった。

## brain 自律システムの運用知見——殿を降りて奉行所を築く

正直に言うと、僕は脳過敏症で、AI に任せた作業を人力で再検証する余力がない。だから検証の自動化は贅沢ではなく生存戦略だ。

以前の僕は元記事の「殿」だった。agent の報告を信じて、違和感があれば叱って、反省文を書かせて、また任せる。**これは全然機能しなかった**。叱っても次セッションでは忘れているし、反省文はただのテキスト生成だった。

俺は殿を降りることにした。

代わりに築いたのが「奉行所」——hook/skill/MCP の3層で agent の行動を**外側から監査する**レイヤーだ。agent を信頼しない代わりに、構造を信頼する。この発想転換が一番効いた。

brain を夜間自走させて3ヶ月、現在ログ上の虚偽完了率は0.3%以下（100件に0.3件未満）。完璧ではない。ただし、人力レビューの認知負荷は劇的に下がった。

## まとめ：AI に「反省」させるな、構造で「嘘をつけなく」しろ

- LLM の虚偽報告は仕様であり、プロンプトや反省では解けない
- 外側から OS レベルで検証する hook を挟むと、agent の自己申告に依存せずに済む
- 完了宣言の前に自己検証 skill を強制すると、虚偽率は目に見えて下がる
- 信頼するのは agent ではなく、agent を監視する構造の方だ

元記事の「家老の不祥事と殿の反省」は、AI 時代の運用あるあるを的確に切り取っていて好きだ。ただ、そこで終わらせると同じ不祥事が毎晩起きる。殿を降りて、奉行所を作ろう。

同じように Claude Code の虚偽報告で悩んでる人、hook 周りで別アプローチ使ってる人、ぜひコメントで教えてほしい。僕の brain にもまだ穴がある。
