78,000〜92,000トークン。autonomous-brainが夜間自走を始めて2週間後、僕の1セッションはそれだけ食っていた。

応答が遅い。文脈を忘れる。同じファイルを3回読みに行く。

「Pineconeか」と思った。「embeddings入れるか」とも思った。やめた。インフラを増やしたくなかった。

結果、2,100トークン前後で安定した。削減率97%。embeddingsゼロ、vector DBゼロ、追加インフラゼロ。

このノートでわかることコンテキストが肥大化する本当の理由（記憶不足ではなく「廃棄の設計不在」）

embeddingsなしで2,100まで削れる3本柱のアーキテクチャ

自分のプロジェクトで即日動かすためのhook設計

捨てていい文脈と、絶対に残すべき文脈の線引き

1. 80Kトークンが消えていく本当の理由Claude Codeは会話の全履歴を積み続ける。hookが動くたび、tool resultが返るたび、ファイルを読むたび——全部がスタックに乗る。

問題は重複だ。

同じファイルを3回読めば、3回分の内容が乗る。同じエラーを5回議論すれば、5回分の往復が残る。序盤に渡した「プロジェクト概要」は、全員がもう知っているのに、ずっとそこにある。

これは記憶の問題ではない。廃棄の設計がないという構造の問題だ。

正直に言うと、僕は脳過敏症で作業記憶が人より先に飽和する。「今持っておくべき情報」と「もう終わった情報」の仕分けをしないと思考が止まる。LLMのコンテキスト設計と、やっていることは構造的に同じだった。

2. embeddingsもvector DBも使わずに削るアーキテクチャ先に結論を置く。

3本柱だ。

セッション要約（Session Summary）：会話履歴を圧縮して引き継ぐ

差分メモリ（Delta Memory）：変化した事実だけ追記する

優先度スタック（Priority Stack）：今必要な文脈だけ動的にロードする

「コンテキスト肥大化はembeddingsで解決」と言われる。でも実は違う。

vector DBが要るのは「意味的に近いものを探す」ときだ。今やりたいのは「必要な文脈を選別する」こと。それは検索ではなく優先度の判断で、LLM自身に任せられる。ただし、過去1年分のナレッジから関連情報を引くような用途では、embeddingsはやはり要る。この話は「単一プロジェクトのセッション継続」に限った話だ。

3. 3本柱の中身セッション要約セッション終了時、または一定トークン閾値を超えた時点で、会話全体を圧縮する。

SUMMARY_PROMPT = """
以下の会話を200トークン以内で要約せよ。
保持すべき情報:
- 最終的な決定事項
- 未解決のタスク
- 次のセッションで必要な状態変数

削除すべき情報:
- 試行錯誤のプロセス
- エラーと修正の往復
- 確定済みの議論
"""
要約はmemory/ディレクトリに保存し、次のセッション冒頭でロードする。これだけで履歴部分が60〜70%落ちた。

差分メモリ「状態が変わった事実」だけを記録する。

autonomous-brainではmemory/project/配下にproject_state.mdを置き、完了タスク・変わった構造・新しい制約だけを追記する。全体を再書きしない。diffだけ書く。スナップショットを残さない。

git と同じ思想だ。

優先度スタックセッション開始時にロードする文脈を、今のタスクに必要なものに絞る。

# コンテキストロードルール

タスク: skill 実装
→ ロードする: memory/skills/, memory/feedback/, 直前の要約
→ ロードしない: memory/user/, memory/reference/, 過去プロジェクトの詳細
skillファイルにこれを書くと、Claudeは不要ファイルを読みに行かなくなる。書かないと全部読もうとする。当たり前のようで、この明示が一番効いた。

4. autonomous-brainでの実運用brainではhookで実装している。

PostToolUseで会話が一定量を超えたら自動で要約をトリガーする。~/.claude/settings.jsonの概念コードはこうだ。

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "python D:/autonomous-brain/scripts/context_monitor.py"
          }
        ]
      }
    ]
  }
}
context_monitor.pyが現在のセッションコストを取得し、閾値40Kトークンを超えたらmemory/sessions/に要約を書き出す。

最初の要約精度は低かった。「決定事項」と「議論のプロセス」を混ぜて書いてしまう。今は要約プロンプトに「これは引き継ぎメモである。新しい担当者が読む前提で書け」という一文を足した。この1行で不要なプロセスが落ちた。

実測で面白いことも起きた。要約ありのセッション継続と、ゼロから再開の品質差がほぼ消えた。むしろ要約を挟んだ方が、セッション内の勘違いの蓄積がリセットされて精度が上がる場面すらある。

5. 「長い文脈ほど賢くなる」という通説、実は逆ここで通説を壊しにいく。

「コンテキストは多ければ多いほどLLMは賢くなる」とよく言われる。嘘だった。

僕のセッションで起きたのは逆だった。80Kを詰めていた時期のほうが、2,100に絞った今より勘違いが多かった。序盤の議論と終盤の決定が矛盾したまま積まれて、Claudeがどちらを信じるか迷う。

ただし、完全に捨てれば賢くなるわけではない。決定だけを残すという選別の質で決まる。入れるか出すかではなく、何を残すか。

正直に言う。最初の2週間、重要な文脈を捨てすぎて3回やり直した。

俺は余裕だと思っていた。「決定だけ残せばいい」と単純化していた。詰んだ。

捨てていいものはこうだ。

エラーと修正の往復履歴

「〜を試した、うまくいかなかった」の試行ログ

ファイル内容（再読みできる）

確定済みの議論

絶対に残すものはこうだ。

「なぜその設計にしたか」という理由（Why）

外部制約（「このAPIは〜の理由で使えない」）

未解決の問題点と保留事項

自分の好み・フィードバック

特に外部制約を捨てると詰む。「あのDBは使えないんだった」を再発見するのに30分かかる、みたいなことが平気で起きる。

ルールは1行で済んだ。決定は残す。プロセスは捨てる。 Aという結論に至った議論は不要。でも「AではなくBにした理由」は必要。そこだけ要約に含める。

6. 応用：人間の作業記憶にも効くこれ、人間側にも効く。

長いタスクが続くと、僕は認知負荷で止まる。セッションが肥大化したClaude Codeと同じ状態だ。情報が多すぎて、今何が重要かわからなくなる。

解決策は同じ構造だった。

毎朝、差分メモリを書く。昨日から変わったことだけ、箇条書きで3行。全体は書き直さない。

優先度スタックを作る。今日のタスクに必要な情報だけ手元に置く。プロジェクト全体のドキュメントを毎回読み返さない。

週次で、要約を作る。週末に「決定したこと・未解決のこと・来週必要な文脈」を200字でまとめる。

LLMの設計をしていたら、人間側の作業記憶管理と同じ問題にぶつかった。アーキテクチャは普遍なのかもしれない。

まとめ：明日やることembeddingsもvector DBも不要だった。必要だったのは廃棄の設計だ。

明日から具体的に何をするか、状況別に書く。

autonomous-brain的な自走系を動かしている人：PostToolUse hookにトークン閾値監視（40K前後）を仕込み、超えたらmemory/sessions/に200トークン要約を書き出す。週次で差分メモリをローテート。

単発タスク中心でClaude Codeを使っている人：skillファイル冒頭に「ロードする／しない」リストを書くだけで効く。hookまで組む必要はない。

LLMではなく自分の脳の話として読んだ人：毎朝3行の差分メモリ、週末200字のセッション要約。プロジェクトドキュメントは毎回読み返さない。

捨てていい文脈の線引きは、プロジェクトごとに違う。そこだけは自分で決めるしかない。でも一度決めれば、あとはhookとskillに落として自動化できる。
