ファインチューニングに半日溶かした経験がある人なら、この1行でゾクッとするはずだ。
**Claude Code の skill 1個で、データ準備→学習→評価→デプロイを1会話完走させる実装が Reddit に投下された。**

通常なら CLI 往復20回超、Notebook のセル管理に数時間、GPU 確保でもうひと手間。それが skill.md 1枚と hook の組み合わせで吸収される設計だ。

データ整形。Docker。GPU 確保。全部 skill に飲み込まれた。

正直に言うと、これを見た瞬間、俺は自分の brain(夜間自走する自律AIシステム)に組み込んだ skill 27個の設計を全部見直した。

## 何が公開されたのか:ML基盤を駆動するskillの構造

r/ClaudeAI に投稿されたこの skill は、ML基盤をまるごと会話から駆動するものだ。「このデータでファインチューニングして」と投げると、後続が連鎖する(skill の内部実装は未検証、以下は brain で類似構成を組んだ経験からの推測を含む)。

僕が brain で似た設計を組んだとき、工程はこう分かれた。

- データ取得・整形(schema 検証とトークナイズ)
- 学習ジョブの起動(GPU 確保・Docker・環境変数注入)
- 学習中のメトリクス監視(loss 曲線・GPU 使用率)
- 評価(ベンチ実行とスコア集計)
- デプロイ(モデル公開・推論エンドポイント登録)

これを skill 1枚の markdown に全部書くのは無理だ。だから **skill の内部で「サブ skill の呼び出し」と「hook によるイベント駆動」が発生している**と推測するのが自然になる。

## skill / hook / MCP の役割分担

1会話で完走するためのキモは、役割を3層に分けることだ。

- **skill**: ユーザーの意図解釈と、サブタスクへの分解。プロンプト設計と成果物フォーマットの責務。
- **hook**: PreToolUse / PostToolUse で学習ジョブの起動状態を検証。GPU 確保失敗、OOM、データ形式不正を**会話に持ち込まない**ためのガード。
- **MCP**: GPU サーバーや学習基盤(runpod、modal、sagemaker 等)への実接続。状態取得と実行の責務。

この分離ができていない skill は、だいたい「長いプロンプトで全部やらせる」構造になり、ジョブが転けると会話ログが汚れて二度と辿れない。

brain で痛い目を見たことがある。hook で検証せず skill から直接 MCP を叩いたら、学習ジョブの失敗メッセージが会話中に約3000行流れてきて、Claude が完全に迷子になった。それ以来、**hook でエラーを握り潰して summary だけ会話に返す**設計に切り替えた。

## 「会話1本でMLは回せない」と言われてきたが、実は

ここが本記事の山場だ。

MLOps界隈では長らく「会話ベースの自動化はML基盤には向かない」と言われてきた。理由は3つある。

1. 学習ジョブは長時間(数時間〜数日)で、会話の文脈が切れる
2. GPU 確保やデータ移動は副作用が大きく、LLM に任せるとコストが吹き飛ぶ
3. 評価の判断基準がドメイン固有で、汎用プロンプトでは書ききれない

**これが今回の skill で部分的に崩れた。**

長時間問題は「hook で状態を永続化 → 再開時に文脈を復元」で回避できる。コスト問題は「MCP 側で予算上限ガードを実装」で封じ込められる。評価問題は「skill 内に評価ルーブリックを外部ファイル参照で持たせる」で分離できる。

**ただし**、完全に崩れたわけではない。ドメイン特化の前処理、分散学習のチューニング、RLHF のような長サイクル工程は依然として人間の介在が必要だ。今回の skill が通じるのは、おそらく「データが既にきれい」「学習ジョブが単機 GPU で収まる」「評価が標準ベンチで済む」範囲に限られる。それでも価値はデカい。個人開発者の90%のケースをカバーするからだ。

## brainに移植するなら:分割と安全装置の設計図

移植するならこう組む。

- skill を3つに分ける: `finetune-prepare` / `finetune-run` / `finetune-eval`
- hook は PostToolUse で**学習ジョブ ID を永続化**。会話が切れても `brain resume <job_id>` で再開できる
- MCP は GPU 予算上限を**環境変数で強制**。超過したらジョブごと kill
- skill.md は**1枚400行以内**に収める。超えたら必ず分割

正直に言うと、今の brain はまだここまで綺麗に切れていない。skill 27個のうち、ML ジョブ系は hook での永続化が甘く、セッション断でジョブ状態が孤児化するケースがある。

読んだ。気づいた。直した。

今回の skill は俺の brain の穴を教えてくれた。感謝している。

## 真似すべき3パターンと、真似してはいけない1つの罠

真似すべき:
1. **skill 内で必ずサマリだけを返す**。ログ全文を流さない。
2. **hook での事前検証**。GPU 空き・データ形式・API 鍵の有無を会話前に潰す。
3. **MCP 側に予算ガード**。LLM 側の理性に頼らない。

真似してはいけない罠:
- **skill だけで全部やる設計**。hook と MCP を使わずプロンプトに詰め込むと、初回は動くが2回目以降で必ず崩れる。skill は司令塔、実行は別レイヤー。この線を引けないと、長時間ジョブで必ず詰む。
